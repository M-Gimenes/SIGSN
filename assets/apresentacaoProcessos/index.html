Fluxo de Transação

Operação atômica. O create executa dentro de uma transação Sequelize: as validações usam a mesma conexão da gravação, e qualquer falha de regra dispara ValidationError e desfaz a inserção.

§Sequência da operação

i
beginAbrir transação
sequelize.transaction abre BEGIN no SQLite.
ii
readBuscar guia + validar campos
Guia.findByPk · validarCampos(Agendamento, dados).
iii
ruleAplicar Regra 1 e Regra 2
disponibilidade do guia · limite por turno.
iv
commitPersistir e retornar
Agendamento.create · findByPk com include · COMMIT.

§AgendamentoService.create

// AgendamentoService.js
static async create(req) {
  const { dataVisita, valorVisita, observacoes,
          guiaId, caravanaId } = req.body;
  const dados = { dataVisita, valorVisita,
                  observacoes, guiaId, caravanaId };

  return sequelize.transaction(async (transaction) => {
    await validarRegrasDeNegocio(
      dados, null, transaction);
    const { id } = await Agendamento.create(
      dados, { transaction });
    return Agendamento.findByPk(
      id, { include, transaction });
  });
}

// → orquestrador validarRegrasDeNegocio:
//   carrega Guia, valida campos, deriva turno,
//   chama Regra 1 e Regra 2 (próximas folhas).

Regras de Negócio

Duas regras protegem a integridade da agenda. Ambas operam dentro da mesma transação de criação/atualização — qualquer violação aborta a operação antes do INSERT.

i
Regra 1 · Disponibilidade do Guia
O guia selecionado deve ter disponibilidade compatível com o turno derivado da hora da visita (MANHA, TARDE, NOITE).
ii
Regra 2 · Limite por turno
Não pode haver mais que 3 agendamentos para a mesma data dentro do mesmo turno (excluindo o próprio registro em update).

§Função orquestradora

async function validarRegrasDeNegocio(dados, excludeId, transaction) {
  const erros = await validarCampos(Agendamento, dados);

  const guia = await Guia.findByPk(
    dados.guiaId, { transaction });
  if (!guia) erros.push('Guia não encontrado.');

  if (erros.length) throw new ValidationError(erros);

  const visitDate = new Date(dados.dataVisita);
  const turno = getTurnoFromDate(visitDate);

  await validarDisponibilidadeGuia(guia, turno);
  await validarLimiteAgendamentosPorTurno(
    visitDate, turno, excludeId, transaction);
}

Implementação da Regra 1 · Disponibilidade do Guia

Regra 1. Compara em memória o turno disponível do guia com o turno derivado da hora da visita. Não há SQL adicional além do findByPk já feito antes — a regra é puramente de domínio.

§AgendamentoService.js · Regra 1

// Regra 1: guia precisa estar disponível no turno escolhido.
async function validarDisponibilidadeGuia(guia, turno) {
  if (guia.disponibilidade !== turno) {
    throw new ValidationError(
      `O guia "${guia.nome}" não tem disponibilidade
       compatível com o horário selecionado.`
    );
  }
}

// agendamentoTurno.js — turno derivado da hora local
export function getTurnoFromDate(date) {
  const d = date instanceof Date ? date : new Date(date);
  const h = d.getHours();
  if (h >= 6  && h < 12) return 'MANHA';
  if (h >= 12 && h < 18) return 'TARDE';
  return 'NOITE';
}

⚙SQL emitido pela busca prévia do guia

A regra em si não dispara consulta — só compara strings. O SELECT abaixo já foi emitido por Guia.findByPk antes da chamada:

SELECT id, nome, especialidade,
       disponibilidade, status
FROM   guias
WHERE  id = $guiaId
LIMIT  1;

Implementação da Regra 2 · Limite por turno

Regra 2. Conta agendamentos no mesmo dia e mesmo turno; em update, exclui o próprio id. O Sequelize aplica date(data_visita) no SQL para comparar apenas a parte da data.

§AgendamentoService.js · Regra 2

const MAX_AGENDAMENTOS_POR_TURNO = 3;

async function validarLimiteAgendamentosPorTurno(
    visitDate, turno, excludeId, transaction) {

  const dayStr = formatLocalDateYYYYMMDD(visitDate);

  const agendamentosNoDia = await Agendamento.findAll({
    attributes: ['id', 'dataVisita'],
    where: sequelize.where(
      sequelize.fn('date', sequelize.col('data_visita')),
      dayStr),
    transaction,
  });

  const totalNoTurno = agendamentosNoDia.filter((row) =>
    Number(row.id) !== Number(excludeId) &&
    getTurnoFromDate(row.dataVisita) === turno
  ).length;

  if (totalNoTurno >= MAX_AGENDAMENTOS_POR_TURNO)
    throw new ValidationError(
      `Não é permitido mais de ${MAX_AGENDAMENTOS_POR_TURNO}
       agendamentos para a mesma data no mesmo turno.`);
}

⚙SQL Resultante (SQL tradicional)

A combinação sequelize.where + sequelize.fn('date', col) gera um filtro pela parte da data, ignorando a hora:

SELECT id, data_visita
FROM   agendamentos
WHERE  date(data_visita) = '2026-05-12';

Fluxo de Transação

Curto-circuito útil. Quando o status NÃO é 'ativo', as regras 1 e 2 são puladas — projeto concluído ou suspenso não disputa cota. Em update, o id atual é excluído da contagem para não competir consigo mesmo.

§Sequência da operação

i
beginAbrir transação
Promise.all carrega Grupo + Coordenador em paralelo.
ii
guardStatus ≠ 'ativo' ?
Se sim, retorna sem aplicar regras de limite.
iii
ruleRegra 2 antes da Regra 1
Mensagem por coordenador é mais específica.
iv
commitcreate + findByPk + COMMIT
Retorna projeto com grupo, coordenador e observações.

§ProjetoService.create + validar

static async create(req) {
  const dados = { ...req.body, status: req.body.status ?? 'ativo' };
  return sequelize.transaction(async (transaction) => {
    await validarRegrasDeNegocio(dados, null, transaction);
    const { id } = await Projeto.create(dados, { transaction });
    return Projeto.findByPk(id, { include, transaction });
  });
}

async function validarRegrasDeNegocio(dados, excludeId, transaction) {
  const erros = await validarCampos(Projeto, dados);
  const [grupo, coordenador] = await Promise.all([
    GrupoDePesquisa.findByPk(dados.grupoDePesquisaId, { transaction }),
    Coordenador.findByPk(dados.coordenadorId, { transaction }),
  ]);
  if (!grupo) erros.push('Grupo de Pesquisa não encontrado.');
  if (!coordenador) erros.push('Coordenador não encontrado.');
  if (erros.length) throw new ValidationError(erros);

  if (dados.status !== 'ativo') return;  // curto-circuito

  await validarLimiteProjetosPorCoordenador(
    dados.coordenadorId, excludeId, transaction);          // Regra 2
  await validarLimiteProjetosAtivos(excludeId, transaction);  // Regra 1
}

Regras de Negócio

Duas regras de capacidade. A regra global (10 projetos ativos no observatório) e a regra individual (2 ativos por coordenador) operam em conjunto. A função exclude remove o próprio id da contagem em operações de update.

i
Regra 1 · Limite global
Não pode existir mais que 10 projetos ativos simultaneamente em todo o observatório.
ii
Regra 2 · Limite por coordenador
Um coordenador pode ser responsável por no máximo 2 projetos ativos simultaneamente.

Implementação da Regra 1 · Limite global de 10 projetos

Regra 1. Usa Projeto.count com filtro status='ativo'. O Sequelize emite um SELECT COUNT(*) — leve, sem trazer linhas. Em update, o id atual é removido com Op.ne.

§ProjetoService.js · Regra 1

async function validarLimiteProjetosAtivos(excludeId, transaction) {
  const total = await Projeto.count({
    where: {
      status: 'ativo',
      ...exclude(excludeId),
    },
    transaction,
  });

  if (total >= MAX_PROJETOS_ATIVOS) {
    throw new ValidationError(
      `Não poderá existir mais que ${MAX_PROJETOS_ATIVOS}
       projetos ativos simultaneamente.`
    );
  }
}

⚙SQL Resultante (SQL tradicional)

Após a expansão do where e do Op.ne em update, o SQL é uma contagem direta:

SELECT COUNT(*) AS count
FROM   projetos
WHERE  status = 'ativo'
  AND  id != $excludeId;   -- ausente em create

Implementação da Regra 2 · Máx. 2 ativos por coordenador

Regra 2. Mesma estrutura da Regra 1, mas filtrando coordenadorId. Composição de where com spread permite reaproveitar o helper exclude sem duplicar código.

async function validarLimiteProjetosPorCoordenador(
    coordenadorId, excludeId, transaction) {

  const total = await Projeto.count({
    where: {
      status: 'ativo',
      coordenadorId,
      ...exclude(excludeId),
    },
    transaction,
  });

  if (total >= MAX_PROJETOS_ATIVOS_POR_COORDENADOR) {
    throw new ValidationError(
      `Um coordenador só poderá ser responsável por
       no máximo ${MAX_PROJETOS_ATIVOS_POR_COORDENADOR}
       projetos com status Ativo.`
    );
  }
}

static async create(req) {
  const { dataObservacao, descricao, instrumentoUtilizado,
          projetoId, constelacaoId } = req.body;
  const dados = { dataObservacao, descricao,
                  instrumentoUtilizado, projetoId, constelacaoId };

  return sequelize.transaction(async (transaction) => {
    await validarRegrasDeNegocio(dados, null, transaction);
    const versaoObservacao = await
      calcularProximaVersao(dados, null, transaction);
    const { id } = await Observacao.create(
      { ...dados, versaoObservacao }, { transaction });
    return Observacao.findByPk(id, { include, transaction });
  });
}

const escopoVinculo = ({ projetoId, constelacaoId }, excludeId) => ({
  projetoId,
  constelacaoId,
  ...(excludeId != null && { id: { [Op.ne]: excludeId } }),
});

Regras de Negócio

Limite + versionamento, juntos. A Regra 1 protege a integridade (no máximo duas observações por dia no mesmo par). A Regra 2 substitui input do usuário pelo cálculo automático de versão sequencial.

i
Regra 1 · Limite diário
Não pode haver mais de 2 observações da mesma constelação no mesmo dia associadas ao mesmo projeto.
ii
Regra 2 · Versão automática
A versão é sequencial por (projeto, constelação): novo registro recebe MAX(versao) + 1. Se mudar o vínculo em update, recalcula.

§Função orquestradora

async function validarRegrasDeNegocio(dados, excludeId, transaction) {
  const erros = await validarCampos(Observacao, dados);
  if (!dados.constelacaoId)
    erros.push('Constelação observada deve ser informada.');

  const [projeto, constelacao] = await Promise.all([
    Projeto.findByPk(dados.projetoId, { transaction }),
    dados.constelacaoId &&
      Constelacao.findByPk(dados.constelacaoId, { transaction }),
  ]);
  if (!projeto) erros.push('Projeto não encontrado.');
  if (dados.constelacaoId && !constelacao)
    erros.push('Constelação não encontrada.');

  if (erros.length) throw new ValidationError(erros);

  await validarLimiteDiario(dados, excludeId, transaction);
}

Implementação da Regra 1 · Limite diário por par

Regra 1. Conta observações no escopo (projeto, constelação, dia). O spread de escopoVinculo evita repetição entre regra 1 e versionamento.

§ObservacaoService.js · Regra 1

const MAX_POR_DIA = 2;

async function validarLimiteDiario(dados, excludeId, transaction) {
  const total = await Observacao.count({
    where: {
      ...escopoVinculo(dados, excludeId),
      dataObservacao: dados.dataObservacao,
    },
    transaction,
  });

  if (total >= MAX_POR_DIA) {
    throw new ValidationError(
      `Não é permitido registrar mais de ${MAX_POR_DIA}
       observações da mesma constelação no mesmo dia
       associadas ao mesmo projeto.`
    );
  }
}

Implementação da Regra 2 · Versão automática

Regra 2. Calcula a próxima versão como MAX(versaoObservacao) + 1 dentro do escopo (projeto, constelação). O Sequelize gera SELECT MAX(...) e o serviço sobrepõe o valor que iria ao banco.

§ObservacaoService.js · Regra 2

// Regra 2: versão é sequencial por (projeto, constelação).
async function calcularProximaVersao(dados, excludeId, transaction) {
  const max = Number(await Observacao.max(
    'versaoObservacao', {
      where: escopoVinculo(dados, excludeId),
      transaction,
    }));
  return Number.isFinite(max) ? max + 1 : 1;
}

// No update: só recalcula se o vínculo mudou.
const mudouVinculo =
  dados.projetoId    !== observacao.projetoId ||
  dados.constelacaoId !== observacao.constelacaoId;
const versaoObservacao = mudouVinculo
  ? await calcularProximaVersao(dados, id, transaction)
  : observacao.versaoObservacao;
